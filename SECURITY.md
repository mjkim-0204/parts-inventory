# 보안 체크리스트

이 저장소는 **공개(public) 저장소**입니다. 커밋되는 모든 것은 이미 "공개"라고 가정하고 작업하세요.

## 0. 기본 원칙 (작업 전에 항상 떠올릴 것)

1. **브라우저에 내려가는 코드(`index.html` 등 클라이언트 JS)에는 비밀번호·API 키·원가 데이터를 절대 넣지 않는다.** 우클릭 → 소스 보기로 누구나 읽을 수 있음.
2. **권한 검사는 항상 서버(Apps Script 함수) 쪽에서 한 번 더 확인한다.** 클라이언트가 "나 로그인했음"이라고 보내는 값은 조작 가능하므로 그대로 믿지 않는다.
3. AI(바이브 코딩)는 시키지 않으면 보안을 최적화하지 않는다. 기능을 완성한 뒤 **"보안 관점에서 검토해줘"를 별도로 요청**하는 걸 습관화한다.

## 1. 커밋 전 체크리스트

- [ ] API 키, 배포 URL/ID, 비밀번호, 토큰이 코드·설정·메모 파일(`.txt` 포함)에 평문으로 들어있지 않은가?
- [ ] 새로 추가한 파일이 `.gitignore` 대상이어야 하는 건 아닌가? (사내 메모, 캡처, 접속 정보 등)
- [ ] `git status` / `git diff`로 실제 올라가는 내용을 눈으로 확인했는가?

## 2. 이 프로젝트 특화 리스크

- **Apps Script 배포 액세스 권한**: "모든 사용자"가 아니라 "조직 내 사용자만"이어야 함. (2026-07 기준 Phase 4 인증 전환 전이라 임시로 "모든 사용자"로 열려있는 **알려진 리스크** — Req 14 완료 시 반드시 재배포로 좁힐 것)
- **스프레드시트(DB 역할)**: 원가·공급처 단가 등 민감한 값이 포함되어 있음을 항상 인지
- **Role 구현 시(Worker/Operator, Req 15)**: 프론트에서 버튼/메뉴 숨김만으로 끝내지 말고, Apps Script 함수 내부에서도 호출자의 실제 권한을 재확인할 것

## 3. 새 API/엔드포인트를 만들 때

- 입력값을 검증하는가? (빈 값, 이상한 문자열, 범위 밖 숫자 등)
- 에러 메시지가 내부 구조(시트 이름, 파일 경로, 스택 트레이스)를 그대로 노출하지 않는가?
- 공개 호출이 가능한 엔드포인트라면 과도한 반복 호출에 대한 방어가 있는가?

## 4. 비밀이 유출됐을 때 대응 순서

1. **재발급/재배포부터** — URL·키 자체를 무효화하는 게 최우선. 히스토리 정리보다 급함.
2. 그 다음 git 히스토리 정리 (선택 사항, `git filter-repo`/BFG + force-push 필요 — 실행 전 반드시 확인)
3. 재발 방지책 하나 추가 (`.gitignore` 규칙 보강, GitHub Secret scanning/Push protection 활성화 등)
